# DDD 학습 노트: 도메인 인터뷰의 중요성

## 1. 요구사항 인터뷰 = 실무에서 고객 인터뷰

개발 전 질문-응답 과정이 실무에서는 이런 형태로 일어난다:

| 인터뷰 질문 | 실무 대응 |
|---|---|
| "사용자 역할이 뭔가요?" | 고객 미팅에서 요구사항 도출 |
| "노쇼 정책은?" | 비즈니스 규칙 확인 |
| "1단계 범위는?" | 스코프 조율 / MVP 정의 |
| "수락 기준 맞나요?" | 인수조건 합의 |

고객이 "예약 시스템 만들어주세요"라고 했을 때, 바로 코딩하면 **거의 반드시** "그게 아니라..."가 나온다.

"1시간 고정인가요?", "노쇼 시 회차 차감인가요?" — 이런 걸 안 물어보면 개발자가 **추측으로 결정**하게 되고, 나중에 뒤엎어야 한다.

## 2. DDD에서 이 과정이 특히 중요한 이유

DDD의 핵심 전제: **"도메인 전문가와 개발자가 같은 언어(Ubiquitous Language)를 쓴다"**

```
고객: "노쇼 2번이면 1회 깎아요"
  ↓ 이 대화에서 나온 용어가 그대로 코드가 됨
코드: NoShowPolicy.Apply(unprocessedCount: 2) → deduct 1 session
```

만약 고객 인터뷰 없이 개발자가 혼자 `penalty_flag = true` 같은 코드를 만들면, **도메인 언어와 코드가 괴리**된다. 나중에 고객이 정책 변경을 요청하면 "이 코드가 그 규칙이었나?" 부터 추적해야 한다.

DDD에서 이 인터뷰 과정을 **Event Storming** 또는 **Domain Discovery**라고 부르고, 설계의 출발점이다.

## 핵심 교훈

> 코드를 잘 짜는 것보다 "뭘 짜야 하는지" 정확히 아는 게 먼저다. 이 순서를 지키는 게 시니어와 주니어의 가장 큰 차이 중 하나다.

## SlotSync 프로젝트에서의 적용

이 프로젝트의 딥인터뷰(10라운드)에서 도출된 핵심 도메인 규칙:
- 트레이너별 가변 세션 시간 → 슬롯 충돌 검증 복잡도 증가
- 노쇼 2회 누적 시 1회차 차감 → NoShowPolicy 도메인 정책
- 월 3회 변경 제한 → 변경 정책 Specification 패턴
- 24시간 전 취소 가능 → CancellationPolicy 도메인 정책

이 규칙들이 인터뷰 없이는 개발자의 추측이 되었을 것이다.

## 3. Aggregate란 무엇인가

### 핵심 한 문장

> Aggregate = "이 데이터는 반드시 함께 일관성을 유지해야 한다"의 경계선

### 비유: 파일 캐비닛

헬스장에 회원 정보 파일 캐비닛이 있다고 상상하면:
- **같은 서랍 안의 것들** → 반드시 함께 일관성 유지 (하나의 Aggregate)
- **다른 서랍** → 독립적으로 변경 가능 (별도 Aggregate)

### 3가지 규칙

1. **Aggregate Root**: 서랍의 "대표". 외부에서는 이 대표를 통해서만 접근
2. **불변식(Invariant)**: 서랍 안에서 "항상 참이어야 하는 규칙"
3. **트랜잭션 경계**: 하나의 Aggregate 변경은 하나의 트랜잭션

### 나쁜 설계 vs 좋은 설계

```
[나쁜 설계] — 전부 하나의 Aggregate
Trainer 안에 프로필 + 스케줄 + 예약 목록 전체
→ 트레이너 이름 바꾸는 것도 예약 락에 걸림

[좋은 설계] — 관심사별 Aggregate 분리
Trainer(프로필/세션길이) | Reservation(시간/상태) | Member(정보/노쇼)
→ 각자 독립적으로 변경 가능, 자기 불변식만 책임
```

### 불변식으로 경계를 결정한다

Aggregate 경계는 "뭘 묶고 싶은가"가 아니라 **"어떤 규칙이 깨지면 안 되는가"**로 결정한다.

| 불변식 | 어디서 보장? |
|---|---|
| 같은 트레이너+겹치는 시간에 중복 예약 불가 | Reservation 생성 시 |
| 잔여 회차 0이면 예약 불가 | Membership |
| 미처리 노쇼 2회 → 1회차 차감 | Member + Membership 연동 |

## 4. "Aggregate 경계를 잘못 잡으면 수습이 어려운가?"

### 맞는 부분: 경계 변경은 비용이 크다

Aggregate 경계 변경 → 테이블 구조, 트랜잭션 범위, API 구조, 테스트 전부 수정.
그래서 앞단에서 도메인 인터뷰에 시간을 쓰는 것이 DDD의 철학이다.

### 틀린 부분: DDD는 이 문제의 대응 장치가 있다

**1) Aggregate 간 느슨한 결합**
```
[강한 결합] Reservation이 Membership 객체를 직접 참조
→ Membership 구조 바뀌면 Reservation도 수정

[느슨한 결합] Reservation은 MembershipId만 들고 있음
→ Membership이 바뀌어도 Reservation은 무관
→ 둘 사이는 Domain Event로 소통
```

**2) Domain Event가 완충재 역할**
```
[직접 호출] reservation.Create() → membership.DeductSession()
→ Membership 구조가 바뀌면 Reservation 코드도 수정

[이벤트 기반] reservation.Create() → ReservationCreated 이벤트 발행
→ Membership이 이벤트를 구독해서 알아서 차감
→ Membership 차감 로직이 바뀌어도 Reservation 코드는 그대로
```

**3) DDD가 인정하는 현실**
> "처음에 틀려도 된다. 단, 틀렸을 때 바꿀 수 있게 만들어라"

이것이 느슨한 결합, 이벤트, Bounded Context 분리를 하는 이유다.

### SlotSync 프로젝트에서의 시사점

- Aggregate 4~5개 수준 → 잘못 잡아도 수습 범위가 작음
- 토이 프로젝트 → 갈아엎는 것 자체가 학습
- 테스트 기반 → 리팩터링 시 안전망 있음
- **"일부러 한번 잘못 잡아보고 왜 문제인지 체감하는 것"도 좋은 학습 방법**

## 5. Aggregate 경계 판단 — 실전 기준

이론적으로 "동시에 변할 수 있는가"가 아니라 **"현실적으로 충돌이 문제를 일으키는가"**로 판단한다.

```
Trainer + 예약 목록 → 예약은 동시에 여러 건 발생 → 충돌 빈번 → 반드시 분리
Member + 노쇼 카운트 → 한 회원이 동시에 2건 노쇼 불가 → 충돌 없음 → 같이 둬도 무방
```

## 6. Aggregate 간 협력 — 트랜잭션 범위

DDD 원칙: 1 트랜잭션 = 1 Aggregate. 하지만 현실적 판단이 필요하다.

- **같은 DB + 모놀리스 + 동시 충돌 없음** → 한 트랜잭션으로 2개 Aggregate 변경해도 괜찮다
- **분산 시스템 or 동시 충돌 빈번** → Domain Event + Outbox 패턴으로 분리 필수

원칙을 아는 것과 언제 타협하는지 아는 것 모두 중요하다.

## 7. Outbox는 Aggregate가 아니라 인프라

Outbox 테이블은 "이벤트를 안전하게 전달하기 위한 배송 상자"로, 도메인 개념이 아닌 인프라 메커니즘이다. 도메인 모델은 Outbox의 존재를 모른다.

구분 기준: **도메인 개념이면 Aggregate, 기술적 해결책이면 인프라.**
